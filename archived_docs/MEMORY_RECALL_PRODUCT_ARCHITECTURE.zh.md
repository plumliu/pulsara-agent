# Pulsara 记忆召回产品化架构方案

_Created: 2026-06-15_

> 本文是 Pulsara 记忆召回的**主张与要义**：我们相信什么、什么最重要、按什么次序推进、什么坚决不做。它只讲立场与要点，不含 DDL、接口签名、文件路径或具体参数——可执行的细节落在实施层，不在本文。
>
> 本文取代已经过时的 `MEMORY_RECALL_FINAL_STORAGE_ARCHITECTURE.zh.md`，并整合：两份框架调研（`MEMORY_RECALL_SYSTEM_SURVEY.zh.md`、`MEMORY_RECALL_MATURE_FRAMEWORKS_SURVEY.zh.md`）、`MEMORY_RECALL_FEASIBILITY_ANALYSIS.zh.md` 的批判，以及产品取向讨论：Pulsara 是产品，不是论文；graph-aware recall 的价值不只是指标提升，也包括解释性、可讲述性、用户感知与系统辨识度。

## 0. 总判断

原“最终整合方案”的**愿景是对的**：

- candidate pool 不是 recall source。
- normal recall 只能使用治理后的 canonical memory。
- transcript/history recall 与 semantic memory recall 必须分开。
- recency 不能创造相关性。
- recall failure 必须结构化，不能让模型幻觉补全。
- projection 是 ephemeral context，不写回 EventLog 用户原文。
- recalled memory 不能被当成本轮新证据原样写回。

但它的**工程排序是错的**：把尚未实现的 recall 写成“只差接线”，把同步图扩展放进 hot path 并预设为核心创新，却没先解决延迟、原子性、评测、类型体系与代码级护栏。

修正后的总路线，是一条**先可靠、再差异化、最后才 fancy** 的演进：

- **可靠底盘**：lexical exact + BM25/FTS 召回，只读治理后的 canonical memory，小预算 projection，显式查询工具，评测护栏，以及防止召回内容被当新事实写回的代码护栏。
- **差异化层**：graph-aware 的重排与**有据可依的解释**——但它必须先有可被解释的事实（关系与状态转换的生产者），且不跌破底盘质量。
- **可选增强**：vector/hybrid/planner，以及把 Oxigraph 作为未来语义图后端——各自凭实测与运营成本赢得默认位置。

一句话：**lexical+BM25 是可靠底盘；graph-aware recall 是产品差异化层；解释必须 100% 有据；Oxigraph 是可插拔后端，不是必选热路径真相源。**

## 1. 产品原则

### 1.1 不是论文式门槛，但解释必须有据

我们不把 graph-aware 层的上线条件写成“必须在 precision@k 上显著打败 lexical baseline”——那是论文门槛，不完全适合产品。用户真正感知的，是“系统为什么召回这条记忆、它由什么证据支撑、是否仍然有效、是否被替代或冲突”。这种**可讲述性本身就是价值**，是 Pulsara 区别于又一个 BM25 memory search 的地方。

但解释是**双刃的**。错误解释比漏召回更伤产品：漏召回只是没找到，用户会再问；而一句假的“这条已被替代”，会训练用户和 agent 去信任一个坏信号，甚至丢弃仍然有效的记忆。所以我们的产品门槛是：

- 质量不跌破可靠底盘（frozen lexical+BM25）。
- 延迟有明确预算、失败可降级回底盘。
- 解释只陈述**已经物化的事实与关系**；拿不出依据时**保持沉默**。“无解释”是合法、常见、正确的输出；强行解释是 bug。

### 1.2 评测是产品护栏，不是论文任务

我们仍然需要 eval——不是为了证明 graph 一定赢，而是为了**防止 fancy 方法悄悄伤害产品**：有没有拉低召回质量、注入无关记忆、让延迟不可接受、错误过滤、或编造解释。门槛必须**可证伪、能落到 CI**：能在本地跑只是工具存在，进入 CI 拦截回退才是门槛。

### 1.3 先可用，再 fancy

v1 不追求“完整图智能”。v1 先让用户体验到：系统真的记得我明确说过的偏好；不会乱召回无关内容；不会把未治理的候选当事实；问“你记得吗”时有工具可查；搜不到时如实说明而不是编。这些做扎实之后，再叠加 graph-aware 的解释与重排。

## 2. 存储姿态

原方案把存储写成三层不可替代的真相源（两个 PostgreSQL 角色 + Oxigraph 语义真相）。修正后的姿态是：

- **Runtime Truth（PostgreSQL）**：发生过什么——会话、运行、回合、事件、工具执行、可重放序列、transcript/history 检索源。它不裁定长期事实真假，可作为 canonical memory 的证据来源。
- **Memory Operational Truth（PostgreSQL）**：记忆运营层——候选池、治理决策日志、artifact、**canonical memory 本身（节点与关系）**、检索索引、召回追踪、保证跨步骤一致的 outbox/reconciliation 状态。
- **GraphStore 适配层**：一个 seam，不是产品价值本身。它可以由 Postgres 关系实现（v1 默认）、由内存视图实现（测试与轻量运行）、或未来由 Oxigraph RDF 实现。

关键主张：**v1 不把 canonical memory 的唯一热路径真相交给进程外的 HTTP 图存储。** canonical memory 以可事务、可索引、可测试的形态存在于 Postgres，至少作为召回底座。这不是否定 typed graph——恰恰相反，这是把 typed graph 的最小可用形态，放进一个能保证一致性、能跑测试的地基上。

产品价值来自**治理化、带状态生命周期、关系感知排序、有据解释的记忆**——这些是本体与排序逻辑的属性，与具体存储引擎无关。Oxigraph 因此被定位为**可插拔的未来后端**（面向 RDF/SPARQL、多跳语义、外部知识融合），而非 v1 必选项。它进入默认路径前，必须先满足异步可取消、查询有界无 N+1、写入原子或可靠补偿、运营与备份清晰——在此之前它是 backend choice，不是产品价值。

## 3. 写入侧主张

保留候选池治理的核心不变量：

- 所有 producer 只写 CandidatePool；只有 governance 能把 candidate 写成 canonical memory。
- canonical write 必经治理写入链路（写入服务 → 写入门 → 证据账本）。
- normal recall 永不读 CandidatePool。

在此之上，我们新增一条更强的写入主张，针对原方案忽视的一致性问题：

> **一次 canonical 写——节点、其必要的证据与关系、治理决策、以及索引/同步记录——必须在同一逻辑提交中保持可恢复一致。**

如果做不到跨存储原子提交，就必须有幂等键、事务性 outbox、对账修复任务、以及“损坏节点”检测。这正是把 canonical memory 放进 Postgres（与候选池、治理日志同库）的红利：最关键的写入可以是一次提交，治理日志才能成为 canonical 状态的可靠镜像，而不是可能与之分叉的旁支。

### 3.1 对现状的诚实表述

我们对今天的写入能力保持精确，避免把蓝图说成已完成：

- 创建记忆时，可产生 **ACTIVE / NEEDS_REVIEW / REJECTED** 三种状态，并落入 canonical store（被 gate 拒绝的节点也会以 REJECTED 落库，并非完全不写）。
- **创建之后的生命周期状态**（STALE / SUPERSEDED / CONTRADICTED / ARCHIVED / DELETED）**目前没有生产者**：没有路径把一个已有节点从 ACTIVE 改成 SUPERSEDED 或 STALE。
- `supersedes` / `contradicts` 等关系目前只存在于本体定义，**没有写入路径**。
- 证据账本只做“创建新节点 + 加证据关系”，**没有更新已有记忆生命周期的语义入口**。

这条诚实表述是后续排序的依据：lifecycle 是一块**尚未动工**的地基，而不是一个待优化的细节。

## 4. 召回侧主张

修正原来“normal recall 只读 Oxigraph”的表述，改为可执行、不自相矛盾的版本：

> normal recall 的**最终事实裁判只认 canonical memory store**；检索索引、lexical、BM25、vector 只负责产出候选 id；候选必须回 canonical store 裁定状态、scope、类型、证据与关系；CandidatePool 永远不是召回源。

由此推出的边界：检索命中不是事实；高分不是事实；transcript 命中不是 semantic memory；`NEEDS_REVIEW` 默认不作事实注入；`REJECTED` 不注入；`SUPERSEDED` 默认不注入（除非用户问历史版本或解释变更）；projection 是 ephemeral，不写回用户原文；召回内容不能被当成新证据写回。

还有一条容易被忽视、但很重要的结论：**因为 canonical store 物理上含有 REJECTED 行，“canonical”这个性质是由召回过滤器定义的，而不是由存储本身定义的。** 过滤不是可选的清理步骤，而是构成性的一步。

### 4.1 两类过滤器必须分清

这是本方案一处关键的诚实化。召回时对 canonical 节点施加的过滤，**性质完全不同的两类，不能混为一谈**：

- **承重过滤器**——今天就在做实活，漏过会立刻出错。例如排除 REJECTED（这些行真实存在于库中）、只保留 ACTIVE（NEEDS_REVIEW 默认仅作诊断）、scope 可达、类型允许、排除本轮 projection 回声。
- **脚手架过滤器**——目前恒真、形同空过滤，因为没有生产者能把节点转进这些状态。例如“排除 superseded / stale / contradicted / archived / deleted”。

主张：**v1 的召回只保证过滤当前已物化的状态与关系，绝不声称已实现自动的 lifecycle 新鲜度。** 脚手架过滤器现在写进逻辑是对的（前向兼容），但我们清楚它要到生命周期生产者就位后（见 §6）才真正生效。把这两类混写，会让人误以为“记得最新、自动淘汰旧版”已经能用——而它还不能。

## 5. v1 召回：lexical exact + BM25/FTS 底盘

v1 既不只做最朴素的 lexical，也不急着引入 embedding。更合理的底盘是**两条互补通道的有界融合**：

- **exact 通道**负责高精度信号——id、别名、scope、类型、工具名、代码标识符、路径、命令、用户原词、项目 token。它解决 BM25 对短字面量、snake_case、路径、端口、命令这类符号不稳定的问题。
- **BM25/FTS 通道**负责自然语言相关性——statement、summary、别名、适用条件、证据摘要、项目元数据。用 token OR 而非脆弱的 AND；有界 top-k；相对分数下限；保留 top-1；零结果给出引导；不做任意的“最近内容”兜底（守 recency 不创造相关性）。

两条通道的结果按 id 去重后**用稳定、可解释的方式融合**（而非把尺度不同的原始分数直接相加），再回 canonical store 过滤（§4.1 的承重过滤器），取小预算的 top-N。这仍是 cheap recall，不是重型 RAG。

底盘的产品承诺很朴素但关键：**用户明确说过的偏好，能在后续对话里被自动召回；无关内容不会被塞进来；问起时有工具可查；查不到时如实说明。**

## 6. Projection 与显式工具

### 6.1 Projection：让自动召回真正发生

v1 的第一件真实产品能力，是让自动 projection 不再空转。projection 是**小预算、可失败降级、不长时间阻塞主流程**的临时上下文。它被围栏标记、带“不要写回”标识、包含记忆 id、召回理由、状态与证据摘要，并明确这是历史记忆、不保证当前世界状态。它默认**不**注入原始 transcript、候选、未治理内容、低置信度待审、无关的近期记忆、或无警告的过期事实。

projection 是 API-call 时的临时上下文，不写回会话历史的用户原文。

### 6.2 显式工具：搜索、读取、解释，与历史检索分开

我们提供少量只读工具，让模型在自动 projection 不够时主动深取：按相关性搜索治理后的记忆、按 id 读取完整记忆节点、（差异化层）查询关联记忆与解释召回理由。工具描述强约束模型：用户问“之前/记得/偏好/决定/上次”时先查记忆；不确定时如实报告“查过但无确认记录”，不要补全；涉及当前文件、配置、外部状态时，召回只是历史线索，必须用当前工具验证。

一条边界要长期坚持：**历史/transcript 检索与语义记忆检索是两个不同入口，不能合并。** 前者回答“我们当时具体说了什么、原话是什么、某次任务怎么收尾”，后者回答“治理后值得长期相信的结论是什么”。两者可互相引用，但不混成一个事实层。

## 7. graph-aware 层：先有可解释的事实，再谈解释

graph-aware 的重排与解释，是 Pulsara 的产品差异化层，不是 v1 阻塞项。它要提供的不是“更高的分”，而是：可解释的召回、证据感知、行动边界匹配、对被替代/被拒记忆的过滤、冲突警告、决策脉络、关联记忆入口——一句话，“系统为什么认为这条记忆现在有用”。

但这一层有一个**被低估的前置**，也是本方案最重要的一处修正：

> **它要解释的事实，必须先有生产者把它写出来。** 没有生产者的关系与状态，不能拿来做产品演示。

### 7.1 生命周期生产者：差异化层真正的第一步

§3.1 已指出：今天没有任何路径把记忆从 ACTIVE 转为 SUPERSEDED/STALE，`supersedes`/`contradicts` 边也无人写。这直接导致两个后果：§4.1 的脚手架过滤器恒真虚设；而“这条旧偏好已被新版替代”“存在冲突需要验证”这类招牌解释，**今天会变成无中生有的编造**。

所以差异化层的真正第一步，不是重排器，而是**生命周期生产者**：

- 把旧节点的状态从 ACTIVE 转为 SUPERSEDED / STALE / CONTRADICTED。
- 写出 `supersedes` / `contradicts` 等正反向关系。
- 发出对应的维护/替代/过期事件。
- 用同一逻辑提交保证跨存储可恢复一致（守 §3 的写入主张）。

这件事比“创建一个新节点”严格更难——它是系统里**第一个真正的多节点原子变更**（新节点入库 + 旧节点改状态 + 关系落地，必须一起成功或一起回滚）。如果不原子，就会出现“新偏好已生效、旧偏好仍标 active、还挂着一条悬空的替代边”这种用户可见的 bug：用户改了偏好 A 为 B，系统却同时召回 A 和 B、都标成 active。

**结论**：生命周期生产者落地之前，差异化层**不得**展示替代/冲突类解释；落地之后，脚手架过滤器才从“恒真占位”升级为“承重”。

### 7.2 解释必须有据、可机器校验

差异化层的解释门槛，比重排更硬：

- 有真实证据关系，才说“有证据支撑”；有真实替代关系，才说“被新版替代”；有真实冲突关系，才说“存在冲突”。
- 只有 lexical/BM25/scope/类型/状态命中时，就只陈述这些真实信号。
- 拿不出依据时**保持沉默**。

为了让“零编造”能真正进 CI，解释不能只是一段自由散文——它必须**结构化**：每条解释带上它所依据的关系或信号，散文只是渲染层。校验器据此逐条核对依据是否真实存在；任何核对不上的，就是编造，必须被拦截。

## 8. ActionBoundary：最高价值，但今天匹配不了

ActionBoundary 是最高价值的记忆类型（它直接改变 agent 行动），但它的“适用条件/不适用条件”目前是**自由文本**，无法便宜可靠地判断是否适用于当前任务。

我们的主张分两步：

- **v1**：ActionBoundary 可被 lexical/BM25 搜到即可。不做昂贵的 LLM 匹配，也不把自由文本当成精确机器规则去解释。
- **后续**：为它增加**结构化触发字段**（触发的工具、动作、文件模式、scope、关键词，以及对应的负向条件），写入时索引，让自动召回能靠索引匹配可靠判断——例如当前任务涉及测试就召回测试相关边界、当前用网络搜索就召回时效/验证边界。

在结构化触发就位前，不声称 ActionBoundary 能被便宜地自动触发。本体扩展遵循既有教训：硬切换、边界处用判别联合、不留兼容垫片。

## 9. 防自我增殖：召回内容不被当新事实写回

仅靠 prompt 里的“不要写回”提示是不够的，必须有**代码级护栏**。系统记录每轮 projection 注入了哪些记忆；在候选生成或治理之前检查：如果一条候选只是本轮召回内容的回声、没有真正的新事实，就拒绝它入池或标记跳过。没有这条护栏，系统会把召回的记忆当成新证据反复写回，形成自我增殖。这是 v1 必做项，不是可选优化。

## 10. 评测护栏与不可侵蚀的底线

评测是产品护栏。它要回答的不是“graph 是否更高分”，而是“fancy 方法有没有伤害产品”：有没有拉低质量、注入无关、错误过滤、超出延迟、或编造解释。

为此我们坚持一条**不可侵蚀的底线**：

> 把 lexical+BM25 底盘的质量**冻结成一个版本化的基线**，任何后续增强（重排、vector、planner）都不得跌破它。

三条要义：

- 这条底线锚定的是 **v1 融合后的真实输出**，不是某个永不单独运行的裸基准。
- 比较是**对固定底线**，不是“相对上一版不更差”——否则每层增强都“只输一点”，底盘会被逐层蚀掉。
- 底线只能由**显式的人工复核**重新冻结，绝不自动刷新；评测集变更必须触发底线复核。

并且，质量底线、零无关泄漏、零编造解释这些硬约束，必须**进入 CI 拦截回退**——能在本地跑只是工具存在，进 CI 才是门槛。

## 11. 推进次序

我们按“先地基、再底盘、后差异化、最后增强”推进，每一阶段都有明确的“做完才进下一步”的判据：

- **地基与安全**：把 canonical 召回底座放进可事务、可索引、可超时的存储；让自动召回受硬超时约束、失败可降级；保证关键写入可恢复一致；时间字段可比较；建立最小评测与 CI 门槛。
- **可靠底盘**：lexical exact + BM25/FTS 召回真正落地；自动 projection 不再空转；显式搜索/读取工具可用；防写回护栏到位。承诺：明确偏好可自动召回、无关不召回、问起可查、查不到不编。
- **耐用索引与可观测**：检索索引可重建、可增量同步、可追踪、可降级；近期召回去重；结构化的不可用与冷却。
- **差异化层**：**先**补生命周期生产者（状态转换 + 正反向关系 + 多节点原子提交），**再**上有据的重排与解释，零编造进 CI。此时脚手架过滤器升级为承重，招牌解释才有真实依据可讲。
- **可选增强**：vector/hybrid/planner，以及把 Oxigraph 作为语义图后端——各自满足“不破坏不变量、不跌破底线、不引入编造、对复杂任务有可感知收益、运营成本可接受”才进默认路径。

更远期，可以从召回使用数据反哺维护（长期召回但从不被用 → 低效用信号；频繁冲突 → 维护候选；过期警告频发 → 验证任务）。但边界不变：召回使用只产生维护候选，canonical 的变更仍走治理与生命周期写入路径，召回绝不旁路修改 canonical。

## 12. 不建议

v1 明确不做：把图扩展放进同步 hot path 并预设为核心创新；把候选池当召回源；没有评测就上复杂重排；只靠 prompt 禁止写回；任意“最近内容”兜底召回；把待审内容当事实自动注入；把历史检索与语义检索混成一个工具；一开始就引入 embedding/LLM planner；以及——在生产者就位前，展示替代/冲突类解释。

## 13. 一句话收束

> Pulsara 记忆召回，先用 lexical exact + BM25/FTS 做可靠底盘；graph-aware 召回作为产品差异化层进入，但必须先有可解释的事实、不跌破底线、有界可降级、解释 100% 有据；生命周期生产者是差异化层真正的第一步；Oxigraph 保留为未来语义图后端，不再作为必选热路径真相源。Pulsara 的价值不只是搜到一段旧文本，而是能告诉用户：这条记忆为什么相关、由什么证据支撑、是否仍然有效、是否被替代、是否与当前任务边界匹配。
